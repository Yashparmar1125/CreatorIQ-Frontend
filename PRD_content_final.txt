1. Purpose & Scope
This document defines the complete frontend requirements for CreatorIQ — an AI-driven YouTube content intelligence platform. It is the single source of truth for the frontend engineering and design teams, covering information architecture, screen-by-screen UI specifications, component library standards, design system tokens, interaction patterns, accessibility requirements, and performance budgets.
This PRD translates the product vision from the main CreatorIQ PRD (v1.0) into actionable, implementable frontend specifications. Every screen, state, component, and interaction required to build the MVP (v1.0) is documented here. v1.1+ features are called out explicitly where they affect architecture decisions.
What This Document Covers
Design system: tokens, typography, color, spacing, motion
Application information architecture and navigation model
Screen-by-screen UI specifications with states (empty, loading, error, populated)
Component library: all reusable components with props and behavior
Form specifications with validation rules
Responsive breakpoints and layout behavior
Accessibility standards (WCAG 2.1 AA)
Frontend performance budgets
API integration contract (what frontend expects from backend)
Developer handoff checklist

5. Screen-by-Screen Specifications

A multi-step wizard shown once on first login. Uses a full-screen overlay layout (no sidebar). Progress shown via step indicators at top. Users can skip all steps except YouTube OAuth (required).

8.2 Validation UX Rules
Validate on blur (not on every keystroke) for performance and UX.
Show inline error below the field in error-600 color. Error icon (AlertCircle) prepended to message.
On form submit: run full validation, scroll to first error, focus first invalid field.
Success confirmation: show success toast (NotificationToast, type: success, 3s auto-dismiss) — never navigate away on success unless explicitly specified.
Destructive actions (disconnect channel, cancel plan): always require ConfirmDialog before submission.

All API calls go through a centralized Axios instance (src/lib/apiClient.ts). This client handles: JWT auth header injection, token refresh on 401, request/response logging in development, global error toasting for 500-level errors.
Base URL: loaded from VITE_API_BASE_URL environment variable. Never hardcode URLs.

9.3 Loading & Error State Handling
Every data-fetching component must handle all four states explicitly. No component may render data without handling the loading and error paths.
State
Required UI Treatment
Loading
Show Skeleton components matching the shape of the loaded content. Never show a blank white area.
Error (API 4xx)
Show inline ErrorCard with specific message from API error.message. Provide retry action where applicable.
Error (API 5xx)
Show generic error toast. Log to error tracking. Provide retry action.
Empty
Show EmptyState component with contextual illustration and action CTA. Never show empty tables or blank panels.
Populated
Render content with Framer Motion fade-in (duration: 250ms, opacity 0→1).
10. Accessibility Requirements (WCAG 2.1 AA)
Every shipped screen must meet WCAG 2.1 Level AA. The following requirements are non-negotiable and will be included in the QA sign-off checklist.
Requirement
Implementation Specification
Color Contrast
All text must meet 4.5:1 contrast ratio against its background. Large text (18pt+): 3:1 minimum. Use Figma Contrast Checker or axe DevTools to verify.
Keyboard Navigation
Full application must be navigable via keyboard alone. Tab order must be logical and match visual order. No keyboard traps.
Focus Indicators
Visible focus ring on all interactive elements. Use brand-700 outline (2px solid) — never remove outlines without a visible alternative.
ARIA Labels
All icon buttons must have aria-label. All form inputs must have associated <label> elements (not just placeholder text).
Screen Reader Support
Charts must include a <table> summary or aria-describedby text alternative. Use role='status' for live metric updates.
Reduced Motion
All Framer Motion animations must check useReducedMotion() hook and disable transitions when true.
Touch Targets
Minimum touch target size: 44×44px on all interactive elements (buttons, links, form controls).
Error Identification
Form errors must be communicated via text (not color alone) and linked to the invalid field via aria-describedby.

11.1 Performance Strategies (Required)
Route-level code splitting: every route must be lazy-loaded via React.lazy() + Suspense
Image optimization: all images served in WebP format with explicit width/height attributes to prevent CLS
Chart libraries: Recharts and Visx must be in separate lazy-loaded chunks
Font loading: use font-display: swap for Google Fonts; preload critical font files
React Query caching: prevent redundant API calls; implement optimistic updates for planner mutations
Memoization: use React.memo and useMemo on expensive chart renders and large list items
12. Testing Requirements
12.1 Unit & Integration Tests (Vitest + RTL)
Coverage target: 80% line coverage across all non-trivial components. Tests must be co-located with source files (ComponentName.test.tsx).
All atom and molecule components: render tests, interaction tests, accessibility checks (jest-axe)
All Zustand stores: action and selector unit tests
All React Query hooks: mock API tests with MSW (Mock Service Worker)
Form validation: test all Zod schemas against valid, invalid, and edge-case inputs
Utility functions in src/lib/: 100% unit test coverage
12.2 End-to-End Tests (Playwright)
Critical user journeys must have E2E tests. These run against a staging environment with seeded test data.
Onboarding flow: complete 5-step wizard including OAuth mock
Trend discovery: filter application, trend card interaction, right panel open
Strategy generation: full session (input → generate → select idea → titles → tags)
Content planner: create slot, drag-and-drop reschedule, delete slot
Analytics: date range change, export CSV
Settings: connect channel, update notifications, change password

14. Developer Handoff & Definition of Done
A screen or component is considered 'done' and eligible for code review only when ALL of the following criteria are met:
14.1 Design Compliance
All design tokens used (no hardcoded colors, fonts, or spacing values)
All four states implemented: loading (skeleton), empty, error, populated
Matches Figma designs at all three breakpoints: mobile (375px), tablet (768px), desktop (1280px)
Hover, focus, and active states visible and correct
14.2 Code Quality
TypeScript: zero 'any' types; all props are fully typed
No inline styles — Tailwind classes or CSS modules only
No hardcoded API URLs or environment values in component code
All useEffect dependencies are correct (no missing or unnecessary dependencies)
React Query used for all data fetching — no raw useEffect fetches
14.3 Accessibility
jest-axe passes with zero accessibility violations in unit tests
Full keyboard navigation works
Color contrast verified with axe DevTools browser extension
All icon buttons have aria-label
14.4 Performance
Lighthouse score ≥ 85 on the affected page (verify in staging)
No unnecessary re-renders (verified with React DevTools Profiler)
No blocking resources in Critical Rendering Path
14.5 Testing
Unit tests written and passing for all new components
E2E test written for any new critical user journey
Coverage remains at or above 80% for the feature module
14.6 Documentation
JSDoc comments on all exported components and hooks
Component added to Storybook (MVP+ — v1.1 scope but story file must be created)
Any non-obvious implementation decisions documented in a code comment
